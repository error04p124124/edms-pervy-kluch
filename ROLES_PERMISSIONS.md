# Система ролей и разграничения доступа

## Обзор

Система ЭДО "Первый ключ" использует ролевую модель разграничения доступа (RBAC - Role-Based Access Control). Каждый пользователь имеет одну из четырех ролей, определяющих его права в системе.

## Роли пользователей

### 1. Администратор (admin)
**Полные права во всей системе**

- ✅ Просмотр всех документов
- ✅ Создание документов
- ✅ Редактирование любых документов
- ✅ Удаление любых документов
- ✅ Регистрация документов
- ✅ Согласование/утверждение документов
- ✅ Управление шаблонами
- ✅ Управление маршрутами согласования
- ✅ Архивирование документов
- ✅ Доступ к отчетам и статистике
- ✅ Управление пользователями

### 2. Делопроизводитель (clerk)
**Управление документооборотом**

- ✅ Просмотр всех документов
- ✅ Создание документов
- ✅ Редактирование любых документов
- ✅ Удаление любых документов
- ✅ Регистрация документов
- ❌ Согласование/утверждение документов (только руководители)
- ✅ Управление шаблонами
- ✅ Управление маршрутами согласования
- ✅ Архивирование документов
- ✅ Доступ к отчетам

**Основная функция:** Ведение документооборота, регистрация входящих/исходящих документов, организация процессов согласования.

### 3. Руководитель (manager)
**Согласование и утверждение документов**

- ✅ Просмотр всех документов
- ❌ Создание документов (через запросы)
- ❌ Редактирование документов (кроме своих черновиков)
- ❌ Удаление документов (кроме своих черновиков)
- ❌ Регистрация документов
- ✅ Согласование/утверждение документов
- ❌ Управление шаблонами
- ❌ Управление маршрутами согласования
- ❌ Архивирование документов
- ✅ Доступ к статистике команды

**Основная функция:** Рассмотрение и согласование документов, принятие решений по документам на своем этапе маршрута.

### 4. Сотрудник (employee)
**Работа со своими документами**

- ❌ Просмотр всех документов (только свои и назначенные)
- ✅ Создание запросов на документы
- ✅ Редактирование своих черновиков
- ✅ Удаление своих черновиков
- ❌ Регистрация документов
- ❌ Согласование документов
- ❌ Управление шаблонами
- ❌ Управление маршрутами
- ❌ Архивирование документов
- ❌ Доступ к общей статистике

**Основная функция:** Создание запросов на документы, просмотр назначенных документов, работа со своими черновиками.

## Статусы документов и права

### Черновик (draft)
- **Создатель:** Может редактировать и удалять
- **Назначенный:** Может редактировать
- **Clerk/Admin:** Полный доступ
- **Manager:** Только чтение
- **Employee:** Только свои

### На регистрации (pending_registration)
- **Clerk/Admin:** Может зарегистрировать или вернуть на доработку
- **Создатель:** Только чтение
- **Остальные:** Только чтение (если назначены)

### Зарегистрирован (registered)
- **Clerk/Admin:** Может редактировать, отправить на согласование
- **Остальные:** Только чтение

### На согласовании (in_review)
- **Согласующие на текущем этапе:** Могут согласовать/отклонить
- **Manager/Admin:** Могут согласовать/отклонить
- **Остальные:** Только чтение

### Утвержден (approved)
- **Clerk/Admin:** Могут архивировать
- **Все:** Только чтение

### Отклонен (rejected)
- **Clerk/Admin:** Могут редактировать, повторно отправить
- **Создатель:** Только чтение
- **Остальные:** Только чтение

### Архив (archived)
- **Все:** Только чтение
- **Clerk/Admin:** Могут восстановить

## Типичные сценарии использования

### Сценарий 1: Сотрудник создает запрос на документ

1. **Employee** создает запрос через "Запросы документов"
2. **Clerk** получает уведомление о новом запросе
3. **Clerk** создает документ на основе запроса
4. **Clerk** назначает документ на сотрудника
5. **Employee** получает уведомление и может просмотреть документ

### Сценарий 2: Делопроизводитель регистрирует входящий документ

1. **Clerk** создает новый документ (статус: draft)
2. **Clerk** заполняет поля документа
3. **Clerk** нажимает "Зарегистрировать" (статус: registered)
4. **Clerk** настраивает маршрут согласования
5. **Clerk** отправляет документ на согласование (статус: in_review)

### Сценарий 3: Согласование документа руководителями

1. Документ со статусом **in_review**
2. **Manager (Этап 1)** получает уведомление
3. **Manager (Этап 1)** рассматривает и согласовывает
4. **Manager (Этап 2)** получает уведомление
5. **Manager (Этап 2)** рассматривает и согласовывает
6. Документ получает статус **approved**
7. **Все участники** получают уведомления

### Сценарий 4: Отклонение документа

1. **Manager** рассматривает документ на согласовании
2. **Manager** выбирает "Отклонить" и указывает причину
3. Документ получает статус **rejected**
4. **Создатель** и **Clerk** получают уведомление с комментарием
5. **Clerk** вносит изменения
6. **Clerk** повторно отправляет на согласование

## Управление ролями

### Назначение роли пользователю

**Через интерфейс администратора:**

1. Перейти в Django Admin: `/admin/`
2. Открыть раздел "Профили пользователей"
3. Выбрать пользователя
4. В поле "Role" выбрать нужную роль:
   - `admin` - Администратор
   - `clerk` - Делопроизводитель
   - `manager` - Руководитель
   - `employee` - Сотрудник
5. Сохранить изменения

**Через код (для разработчиков):**

```python
from accounts.models import UserProfile

# Получить профиль пользователя
profile = UserProfile.objects.get(user__username='ivanov')

# Назначить роль
profile.role = 'manager'
profile.save()
```

### Проверка роли в коде

**Во views:**

```python
from documents.permissions import (
    can_edit_document,
    can_delete_document,
    can_approve_document,
    can_manage_templates,
)

# Проверка с помощью функций
if can_edit_document(request.user, document):
    # Разрешить редактирование
    pass

# Проверка через декоратор
from documents.permissions import clerk_required

@clerk_required
def my_view(request):
    # Доступно только clerk и admin
    pass

# Проверка через mixin (class-based views)
from documents.permissions import ManagerRequiredMixin

class MyView(ManagerRequiredMixin, DetailView):
    # Доступно только manager и admin
    pass
```

**В шаблонах:**

```django
{% load permission_tags %}

{# Проверка прав на редактирование #}
{% user_can_edit_document user document as can_edit %}
{% if can_edit %}
    <a href="{% url 'document_edit' document.id %}">Редактировать</a>
{% endif %}

{# Проверка прав на управление шаблонами #}
{% user_can_manage_templates user as can_manage %}
{% if can_manage %}
    <a href="{% url 'template_list' %}">Шаблоны</a>
{% endif %}

{# Проверка прав на удаление #}
{% user_can_delete_document user document as can_delete %}
{% if can_delete %}
    <button onclick="deleteDocument()">Удалить</button>
{% endif %}

{# Проверка прав на согласование #}
{% user_can_approve_document user document as can_approve %}
{% if can_approve %}
    <form method="post" action="{% url 'approve_document' document.id %}">
        <!-- Форма согласования -->
    </form>
{% endif %}
```

## API функций проверки прав

Все функции находятся в модуле `documents.permissions`:

### `can_edit_document(user, document)`
Проверяет, может ли пользователь редактировать документ.

**Возвращает:** `True` если:
- Пользователь - admin или clerk
- Пользователь - создатель документа и документ имеет статус "draft"
- Пользователь назначен на документ и документ имеет статус "draft"

### `can_delete_document(user, document)`
Проверяет, может ли пользователь удалить документ.

**Возвращает:** `True` если:
- Пользователь - admin или clerk
- Пользователь - создатель документа и документ имеет статус "draft"

### `can_approve_document(user, document)`
Проверяет, может ли пользователь согласовывать документ.

**Возвращает:** `True` если:
- Пользователь - admin или manager

### `can_manage_templates(user)`
Проверяет, может ли пользователь управлять шаблонами.

**Возвращает:** `True` если:
- Пользователь - admin или clerk

### `can_view_all_documents(user)`
Проверяет, может ли пользователь видеть все документы в системе.

**Возвращает:** `True` если:
- Пользователь - admin, clerk или manager
- Для employee возвращает `False` (они видят только свои документы)

## Декораторы для views

Все декораторы находятся в модуле `documents.permissions`:

### `@role_required(*roles)`
Универсальный декоратор для проверки ролей.

**Параметры:** Список ролей (например: `'admin', 'clerk'`)

**Пример:**
```python
from documents.permissions import role_required

@role_required('admin', 'clerk', 'manager')
def sensitive_view(request):
    # Доступно только для admin, clerk и manager
    return render(request, 'template.html')
```

### `@admin_required`
Доступ только для администраторов.

### `@clerk_required`
Доступ для делопроизводителей и администраторов.

### `@manager_required`
Доступ для руководителей и администраторов.

### `@clerk_or_manager_required`
Доступ для делопроизводителей, руководителей и администраторов.

## Mixins для class-based views

Все миксины находятся в модуле `documents.permissions`:

### `RoleRequiredMixin`
Базовый миксин для проверки ролей.

**Атрибут:** `required_roles` - список необходимых ролей

**Пример:**
```python
from documents.permissions import RoleRequiredMixin

class MyView(RoleRequiredMixin, ListView):
    required_roles = ['admin', 'clerk']
    model = Document
```

### `AdminRequiredMixin`
Доступ только для администраторов.

### `ClerkRequiredMixin`
Доступ для делопроизводителей и администраторов.

### `ManagerRequiredMixin`
Доступ для руководителей и администраторов.

### `ClerkOrManagerRequiredMixin`
Доступ для делопроизводителей, руководителей и администраторов.

## Безопасность

### Принципы безопасности

1. **Принцип минимальных привилегий:** Каждая роль имеет только необходимые для работы права.

2. **Защита на уровне views:** Все представления защищены декораторами или миксинами.

3. **Защита на уровне шаблонов:** Кнопки и ссылки отображаются только при наличии прав.

4. **Двойная проверка:** Даже если пользователь попытается напрямую обратиться к URL, права проверяются в view.

5. **Логирование:** Все действия с документами записываются в историю.

### Рекомендации

1. **Не назначайте роль admin без необходимости** - используйте специализированные роли.

2. **Регулярно проверяйте назначенные роли** пользователей.

3. **При увольнении сотрудника** сразу деактивируйте его учетную запись.

4. **Для тестирования** создавайте отдельные тестовые аккаунты с разными ролями.

5. **При добавлении новых функций** всегда добавляйте проверки прав.

## Часто задаваемые вопросы

### Как сотруднику создать документ?

Сотрудники не могут создавать документы напрямую. Они могут:
- Создать запрос на документ через "Запросы документов"
- Делопроизводитель создаст документ на основе запроса

### Может ли руководитель редактировать документы?

Руководитель может редактировать только:
- Свои черновики
- Документы, назначенные на него со статусом "draft"

Для редактирования других документов нужна роль clerk или admin.

### Как изменить роль пользователя?

Только администратор может изменять роли через Django Admin (`/admin/`).

### Что делать, если пользователь не видит нужную кнопку?

Проверьте:
1. Какая роль назначена пользователю
2. Соответствует ли роль требуемым правам
3. Находится ли документ в нужном статусе

### Можно ли назначить пользователю несколько ролей?

Нет, у каждого пользователя только одна роль. Если нужны права из разных ролей, используйте роль admin.

### Как добавить новую роль?

Для добавления новой роли нужно:
1. Добавить роль в `UserProfile.ROLE_CHOICES` в `accounts/models.py`
2. Обновить функции проверки прав в `documents/permissions.py`
3. Добавить соответствующие декораторы и миксины
4. Выполнить миграции: `python manage.py makemigrations && python manage.py migrate`
